# <a name="search"></a>Search

На порталах PowerApps можно выполнять поиск записей по нескольким сущностям с помощью функции глобального поиска на портале. Кроме того, можно выполнять поиск в записях списков сущностей с помощью функции поиска по списку сущностей. 

Функция поиска по списку сущностей на портале использует FetchXML в серверной части для поиска в столбцах, определенных в списке сущностей, и последующего вывода результатов. 

Глобальный поиск использует внешний индекс поиска, основанный на Lucene.Net и используемый для поиска в нескольких сущностях и полях одновременно.

## <a name="global-search"></a>Глобальный поиск

Глобальный поиск порталов позволяет выполнять поиск записей по нескольким сущностям. Он также позволяет выполнять поиск по нескольким столбцам и настраивать, какие столбцы сущности можно найти.

Среди преимуществ глобального поиска есть возможность:
- Поиск совпадений с любым словом в поисковом выражении в любом поле сущности. Соответствия могут включать слова с символами прописью, такие как поток, потоковая передача или потоковая передача.
- Возвращает результаты из всех сущностей, поддерживающих поиск, в одном списке, отсортированном по релевантности, в зависимости от таких факторов, как число совпадений слов, или их сходство друг с другом в тексте.
- Выделение совпадений в результатах поиска.
- Укажите параметры аспектов, которые можно использовать для дальнейшей фильтрации результатов поиска.

В случае глобального поиска, чем выше совпадение, тем выше оно отображается в результатах. Совпадение имеет более высокую релевантность, если больше слов из искомого термина находится в близком расположении друг к другу. Чем меньше объем текста, где обнаруживаются слова поиска, тем выше релевантность. Например, если найти слова для поиска в названии и адресе компании, это может быть более точно, чем те же слова, которые встречаются в большой статье далеко друг от друга. Так как результаты возвращаются в одном списке, можно увидеть сочетание записей, отображаемых один за другим, с выделенным вариантом соответствует. 

В следующих разделах подробно описано, как работает Глобальный поиск на порталах PowerApps, и описаны различные доступные варианты конфигурации.

## <a name="fields-searchable-in-global-search"></a>Поля, поддерживающие поиск в глобальном поиске

Все поля, доступные в представлении, определяемом параметром "Поиск/Индекскуеринаме сайт" для любой сущности, индексируются в глобальном поиске и доступны для поиска. 

Значение по умолчанию для Search/Индекскуеринаме — "Поиск на портале".

Если представление недоступно для какой-либо сущности, оно не индексируется и результаты не отображаются в глобальном поиске.

> [!NOTE]
> Если изменить значение параметра сайт поиска или Индекскуеринаме, необходимо запустить повторное индексирование сборки вручную с помощью шагов, определенных в разделе [Перестроение индекса полного поиска](#rebuild-full-search-index) .

## <a name="entity-specific-handling"></a>Обработка сущностей

- **Вариант**. по умолчанию доступные для поиска варианты находятся в **разрешенном** состоянии, а для поля **опубликовать в Интернете** задано значение **true**. Это поведение можно изменить, обновив представление поиска на портале сущности вариантов и удалив фильтры, доступные в представлении поиска на портале. Однако при удалении этой проверки важно убедиться, что веб-шаблон "служба клиента — случай" изменен, так как этот веб-шаблон позволяет всем пользователям просматривать только те варианты, которые активны и не публикуются в Интернете. Если веб-шаблон не изменяется, варианты будут отображаться в результатах поиска. Однако при их выборе отображается веб-страница сведений о варианте с сообщением об отказе в разрешении.

- **База знаний**: статьи базы знаний доступны только для поиска, если они находятся в **опубликованном** состоянии, а для свойства **внутреннее** поле установлено значение **нет**. Это поведение нельзя изменить. Статьи базы знаний также имеют специальные функции, доступные в результатах поиска следующим образом.

    - **Аспекты**: два специальных аспекта доступны только для статей базы знаний и отображаются, если записи статей базы знаний доступны в результатах поиска.

        - **Аспект рейтингов**. Этот аспект позволяет фильтровать результаты поиска на основе средней оценки статей базы знаний.

        - **Аспект продукта**. Этот аспект позволяет фильтровать результаты поиска на основе продукта, связанного с статьями базы знаний.

    - **Поиск вложений**. Эта функция позволяет выполнять поиск в вложениях или примечаниях, связанных со статьей базы знаний. Это позволяет выполнять поиск по описанию примечания, названию, имени файла вложения и содержимому вложений в примечаниях или вложениях, предоставляемых на портале. Дополнительные сведения: [Поиск в содержимом вложенных файлов](search-file-attachment.md)

## <a name="special-characters-and-syntax-supported-by-search"></a>Специальные символы и синтаксис, поддерживаемые поиском

В рамках глобального поиска на портале поддерживаются разнообразные специальные символы и синтаксис, позволяющие лучше отфильтровать результаты поиска. Эти специальные символы и синтаксисы широко делятся на следующие группы:

- **Term**: каждый запрос, указанный пользователем для поиска, анализируется по терминам и операторам. Ниже приведены типы терминов. 

    - **Одно выражение**. Единственное понятие — одно слово. Например, запрос {Hello World} будет проанализирован на два отдельных термина: "Hello" и "World". Каждый отдельный термин ищется отдельно. Таким образом, в запросе {Hello World} все записи с термином "Hello" или "World" будут отображаться в результатах поиска.

    - **Фразы**. фраза — это группа терминов, заключенных в двойные кавычки (""). Например, запрос {"Hello World"} будет проанализирован как фраза "Hello World". Каждая фраза полностью ищется. Таким образом, в запросе {"Hello World"} все записи, имеющие полную фразу "Hello World", будут отображаться в результатах поиска, и любая запись, которая содержит только "Hello" или "World", отображаться не будет.

    Каждый поисковый запрос может состоять из одного или многих из этих терминов любого типа, Объединенных с помощью логических операторов для создания сложных запросов.

- **Модификаторы Терма**

    - **Поиск с подстановочными**знаками. Существует два типа подстановочных знаков, которые можно использовать в одном термине поисковых запросов (не в запросах фразы): Поиск с использованием подстановочных знаков в одном символе и поиск с использованием подстановочных знаков.

        - **Поиск с подстановочными знаками в одном символе**. чтобы выполнить поиск с использованием подстановочных знаков, используйте символ вопросительного знака (?). Поиск по подстановочным знакам одного символа выполняет поиск терминов, совпадающих с заменой одного символа. Например, для поиска "Text" или "Test" можно использовать поисковый запрос как "Te? t".

        - **Поиск с использованием подстановочных знаков с несколькими символами**. чтобы выполнить поиск с использованием подстановочных знаков, используйте символ звездочки (\*). Поиск по подстановочным знакам с несколькими символами ищет ноль или более символов. Например, для поиска тестов, тестов или тестировщика можно использовать поисковый запрос как "тест *". В середине запроса можно также использовать несколько символов поиска с подстановочными знаками. Например, "Te*t".

        > [!NOTE]
        > - Нельзя использовать * или? символ в качестве первого символа поиска.
        > - Поиск с подстановочными знаками нельзя использовать в запросе фразы. Например, если вы используете Query как "Hell * World", он не будет отображать результаты с текстом "Hello World".

    - **Поиск**с учетом расположения. Поиск по сходству позволяет искать слова, находящиеся на определенном расстоянии друг от друга. Например, если требуется получить результаты для слов «изображение» и «размытие» в пределах 10 слов друг от друга, можно использовать поиск по сходству.
    
        Для поиска с учетом расположения используйте символ тильды (~) в конце запроса. Например, если требуются результаты для слов «изображение» и «размытие», появляющихся в пределах 10 слов друг друга, то запрос будет выглядеть как «размытие изображения» ~ 10.

    - **Повышение уровня термина**. Глобальный поиск обеспечивает релевантность соответствующих документов на основе найденных условий. Чтобы увеличить термин, используйте символ каретки (^) с коэффициентом увеличения (числом) в конце искомого термина. Чем выше коэффициент увеличения, тем более релевантным будет термин.

        Повышение позволяет управлять релевантностью документа путем повышения его срока. Например, если вы ищете интеллектуальный телевизор и хотите, чтобы термин «смарт-канал» был более релевантным, увеличьте его с помощью символа «^» вместе с коэффициентом повышения. Введите: Smart ^ 4 Телевизор. Благодаря этому документы с термином интеллектуальный вид будут более актуальными.

        Вы также можете повысить фразу, как в примере: интеллектуальный телевизор ^ 4 новый телевизор. В этом случае фраза "смарт-телевизор" будет увеличена в сравнении с "новыми ТЕЛЕВИЗОРами".

        По умолчанию коэффициент увеличения равен 1. Хотя коэффициент увеличения должен быть положительным, он может быть меньше 1 (например, 0,2).

- **Логические операторы**. логические операторы позволяют объединять термины с помощью операторов логики. Глобальный поиск поддерживает или, AND, NOT, "+" и "-" как логические операторы.

    > [!NOTE]
    > Логические операторы должны быть написаны в верхнем регистре.

    - **Или**: оператор OR является оператором умножения по умолчанию. Это означает, что при отсутствии логического оператора между двумя терминами используется оператор OR. Оператор OR связывает два термина и находит соответствующую запись, если в записи есть один из терминов. Это эквивалентно объединению с помощью наборов. Символ | | может использоваться вместо слова или. Например, поисковый запрос "интеллектуальный телевизор" (без кавычек) будет искать все записи со словом "интеллектуальный" или "телевизор". Этот запрос также можно написать как «Smart или TV», «Smart | | ТЕЛЕВИЗОР».

    - **И:** Оператор AND сопоставляет записи, в которых находятся оба термина в тексте одного документа. Это эквивалентно пересечением с помощью наборов. Символ & & можно использовать вместо слова и. Например, поисковый запрос "смарт-и телевизор" (без кавычек) будет искать все записи, содержащие слова "интеллектуальный" и "телевизор". Этот запрос также можно записать как "Интеллектуальные & & Телевизор".

    - **Not**: оператор not исключает записи, содержащие термин, после not. Это эквивалентно разнице при использовании наборов. Символ! можно использовать вместо слова NOT. Например, поисковый запрос «смарт-телевизор» (без кавычек) выполнит поиск всех записей, имеющих слово «Smart», но не имеющих слова «TV». Этот запрос также можно написать как "Smart! ТЕЛЕВИЗОР».

    - **Символ "плюс" (+)** . символ "плюс" (+), также известный как обязательный оператор, требует, чтобы термин после символа "+" существовал в записи. Например, поисковый запрос "Smart + TV" будет искать все записи, в которых должно присутствовать слово TV, а также слово Smart. 

    - **Символ минуса (–)** : символ минуса (-), также известный как оператор запрета, исключает документы, содержащие термин после символа «-». Например, поисковый запрос "интеллектуальный телевизор" будет искать все записи, в которых имеется слово Smart, а слово TV не должно присутствовать.

- **Группирование**. Глобальный поиск на портале поддерживает использование круглых скобок для группировки предложений для создания вложенных запросов. Это может быть очень полезно, если требуется управлять логической логикой для запроса. Например, если вы хотите найти все записи, в которых есть хотя бы одно из терминов "HD" или "Smart", но слово TV всегда существует, то запрос можно записать как "(HD или Smart) и телевизор" (за исключением кавычек).

## <a name="liquid-search-tag"></a>Тег поиска жидкостей

Вы можете вызвать глобальный поиск на портале из шаблонов жидкостей с помощью тега сеарчиндекс. Дополнительные сведения: [сеарчиндекс](../liquid/portals-entity-tags.md#searchindex)

> [!IMPORTANT]
> При использовании тега сеарчиндекс аспекты не возвращаются как часть результатов и не могут быть применены в качестве фильтра.

## <a name="update-search-index"></a>Обновить индекс поиска

Обновления индекса поиска на порталах PowerApps происходят автоматически как недействительность кэша. Однако учитывайте следующие важные моменты.

- Для всех сущностей, поддерживающих поиск, должен быть включен флаг метаданных изменения уведомлений. в противном случае портал не будет уведомлен об изменениях, а индекс поиска не будет обновлен.

- Изменение в поиске на портале может занять до 30 минут. Однако 95% изменений будет обновляться в течение 15 минут. Если вложения участвуют, это может занять больше времени в зависимости от размера вложения.

- Рекомендуется перестраивать полный индекс вручную после выполнения многопроцессного переноса или обновления данных в течение короткого промежутка времени. Дополнительные сведения см. в разделе [Перестроение полного индекса поиска](#rebuild-full-search-index).

## <a name="rebuild-full-search-index"></a>Перестроение полного индекса поиска

Перестроение полного индекса поиска требуется каждый раз:

- Вы вносите изменения метаданных в свойства поиска, такие как изменение определенных параметров сайта конкретного запроса или изменение представления поиска сущности, и т. д.
- Выполняется миграция или обновление больших данных.
- Запись веб-сайта, связанная с порталом, изменяется в среде Common Data Service.

Можно также перестроить полный индекс поиска с портала.
1.  Войдите на портал с правами администратора.
2.  Перейдите к URL-адресу следующим образом: `<portal_path>/_services/about`
3.  Выберите **перестроить индекс поиска**.

> [!IMPORTANT]
> Полное перестроение индекса — это очень ресурсоемкая операция, которую не следует выполнять в часы пикового использования, так как это может привести к переходу на портал.

## <a name="remove-an-entity-from-global-search"></a>Удаление сущности из глобального поиска

Иногда может потребоваться полностью удалить определенные сущности из глобального поиска на портале, чтобы клиенты могли быстро получать нужные результаты.

В следующем примере сущность варианта будет удалена из глобального поиска на портале.

### <a name="step-1-block-case-entity-from-getting-indexed"></a>Шаг 1. Блокировка сущности Case из индекса

Чтобы заблокировать сущность вариантов от получения индекса, необходимо переименовать представление сущности варианта, определяющей набор записей, который будет индексироваться порталом (определяется параметром поиска или Индекскуеринаме сайта). По умолчанию именем этого представления является поиск на портале.

1.  Откройте [приложение управления портала](configure-portal.md).

2.  Щелкните значок **параметров** на панели инструментов в верхней части страницы, а затем выберите **Дополнительные параметры**.

2.  Последовательно выберите **параметры** > **Настройка** > **Настройка системы**.

    ![Настройка системы](../media/customize-system.png "Настройка системы")

3.  В диалоговом окне настройки выберите **компоненты** > **сущности** > **регистр** в левой области навигации. 

4.  Разверните сущность **варианта** и выберите **представления**.

5.  Выберите представление **поиска на портале** в списке и откройте его в редакторе представлений.

    ![Представление "регистр"](../media/case-view.png "Представление "регистр"")

6.  В редакторе представлений выберите **вид свойства**.

    ![Редактор представлений](../media/view-editor.png "Редактор представлений")

7.  Переименуйте представление в соответствии с требованием. Убедитесь, что в новом имени нет термина "Поиск на портале".

    ![Просмотр свойств](../media/view-properties.png "Просмотр свойств")

8.  Сохраните изменения и закройте редактор представлений.

9.  Выберите **Опубликовать все настройки**.

10. Перестройте полный индекс, как описано в разделе [Перестроение индекса полного поиска](#rebuild-full-search-index) .

> [!NOTE]
> В этом примере мы делаем изменения в неуправляемом слое, напрямую редактируя представление. Это также можно сделать с помощью управляемого решения.

### <a name="step-2-remove-case-entity-from-the-ui"></a>Шаг 2. Удаление сущности варианта из пользовательского интерфейса

После выполнения действий, описанных в шаге 1, сущность варианта будет остановлена от получения индекса. Чтобы удалить сущность варианта из области пользовательского интерфейса, необходимо изменить параметр сайта, связанный с глобальным поиском на портале. Необходимо изменить следующий параметр сайта:

Поиск и фильтры. сущность case будет удалена из фильтров на странице поиска, а также в поле поиска в заголовке сайта. По умолчанию используется значение: `Content:adx_webpage,adx_webfile;Blogs:adx_blog,adx_blogpost;Forums:adx_communityforum,adx_communityforumthread,adx_communityforumpost;Ideas:adx_ideaforum,adx_idea;Help Desk:incident;Knowledge:knowledgearticle`

Необходимо удалить `Help Desk:incident;` из значения этого параметра сайта, чтобы сущность инцидента была удалена из фильтров, поступающих рядом с полем поиска в пользовательском интерфейсе.

Измененное значение будет следующим:

`Content:adx_webpage,adx_webfile;Blogs:adx_blog,adx_blogpost;Forums:adx_communityforum,adx_communityforumthread,adx_communityforumpost;Ideas:adx_ideaforum,adx_idea;Knowledge:knowledgearticle`

После изменения этого параметра сайта сущность варианта будет удалена из фильтров на странице поиска, а также в заголовке.
