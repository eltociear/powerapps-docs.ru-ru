# <a name="configure-portal-authentication"></a>Настройка проверки подлинности портала

В приложении портала пользователь портала, прошедший проверку подлинности, связывается с контактом или системным пользователем. Конфигурация порталов по умолчанию основана на контактах. Чтобы выполнить вход, для контакта должны быть настроены соответствующие сведения о веб-проверке подлинности. Пользователям портала должны быть назначены веб-роли, чтобы они могли получить более широкие разрешения, чем пользователи, которые не прошли проверку подлинности. Чтобы настроить разрешения для веб-роли, настройте для нее правила управления доступом к веб-страницам доступом к веб-сайтам.

Новейшие средства проверки подлинности портала позволяют пользователям портала выполнить вход со своей выбранной учетной записью на основе локального поставщика членства контакта или с внешней учетной записью на основе [ASP.NET Identity](https://www.asp.net/identity).   

- **Локальная аутентификация**: это стандартная аутентификация на основе форм, которая использует записи контактов среды Common Data Service для аутентификации. Чтобы создать настраиваемый механизм проверки подлинности, можно воспользоваться API ASP.Net Identity и создать настраиваемые страницы входа и инструменты.
- **Внешняя проверка подлинности**: внешняя проверка подлинности предоставляется API ASP.NET Identity. В этом случае управление учетными данными учетной записи и паролем производится сторонним поставщиком удостоверений. Сюда входят поставщики на основе OpenID, такие как Yahoo! и Google, и поставщики на основе OAuth 2.0, такие как Twitter, Facebook и [!INCLUDE[cc-microsoft](../../../includes/cc-microsoft.md)]. Пользователи регистрируются на портале, выбирая внешнее удостоверение, чтобы зарегистрироваться на портале. После регистрации внешнее удостоверение имеет доступ к тем функциям, что и локальная учетная запись. 

При регистрации как с локальной, так и внешней учетной записью можно использовать коды приглашения, а также процесс подтверждения по электронной почте. Кроме того, администраторы порталов могут включить или отключить любое сочетание параметров проверки подлинности в настройках сайта портала.

> [!NOTE]
> Пользователи портала должен иметь уникальный адрес электронной почты. Если две или более записей контактов (включая записи деактивированных контактов) имеют одинаковый адрес электронной почты, эти контакты не смогут пройти проверку подлинности на портале.

## <a name="account-sign-up-registration"></a>Регистрация учетной записи (регистрация)

У администраторов порталов имеется несколько вариантов управления поведением регистрации учетной записи. Открытая регистрация — это наименее ограничивающая конфигурация регистрации, в которой портал позволяет регистрировать учетную запись пользователя, просто указав идентификатор пользователя. Альтернативные конфигурации могут требовать ,чтобы пользователь предоставил код приглашения или действительный адрес электронной почты для регистрации на портале. Независимо от конфигурации регистрации, локальные и внешние учетные записи участвуют в бизнес-процессе регистрации на равных правах. То есть, пользователи могут выбрать, какой тип учетной записи они хотят зарегистрировать.

## <a name="open-registration"></a>Открытая регистрация

Во время регистрации пользователь имеет возможность выбрать создание локальной учетной записи (указав имя пользователя и пароль) или выбрать внешнее удостоверение из списка поставщиков удостоверений. Если выбрано внешнее удостоверение, пользователь должен выполнить вход через выбранного поставщика удостоверений, чтобы доказать, что он является владельцем этой внешней учетной записи. В обоих случаях пользователь сразу же регистрируется на портале и проходит проверку подлинности. Новая запись контакта создается в среде Common Data Service при регистрации.

Если включена открытая регистрация, пользователям не требуется предоставлять код приглашения, чтобы завершить процесс регистрации.
