# <a name="quick-start-for-using-an-existing-environment-to-validate-your-legacy-web-client-app-with-the-unified-interface"></a>Краткое руководство по использованию существующей среды для проверки старого приложения веб-клиента с единым интерфейсом
Этот раздел краткого руководства демонстрирует, как использовать существующую среду для создания приложения единого интерфейса на основе текущей конфигурации или решения по умолчанию. Это позволит вам изучить и протестировать единый интерфейс во время запуска существующих приложений веб-клиента параллельно. Пользователь может затем переключаться между средами для просмотра бок о бок. 

> [!VIDEO https://www.microsoft.com/videoplayer/embed/RE3JzyI]

Похожие инструкции, которые показывают порядок создания новой среды песочницы для изоляции тестирования и просмотра только единого интерфейса, см. в разделе [Краткое руководство по переводу старого приложения веб-клиента Dynamics 365 на единый интерфейс](transition-web-app.md).

> [!IMPORTANT]
>  Для сред с приложениями Dynamics 365 Field Service или Dynamics 365 Project Service Automation см. раздел [Приложения Dynamics 365](transition-web-app.md#dynamics-365-apps).

## <a name="prerequisites"></a>Необходимые компоненты 
- Существующее старое приложение веб-клиента Dynamics 365 Sales или Service. 
- Хотя не обязательно, но рекомендуется использовать непроизводственную среду, чтобы проверить приложение. Дополнительные сведения: [Управление экземплярами в песочнице](/dynamics365/customer-engagement/admin/manage-sandbox-instances) 

## <a name="overview"></a>Обзор 
Этот раздел предназначен для существующих клиентов, которые сейчас используют старые приложения веб-клиентов, которым нужно спланировать и выполнить переход на единый интерфейс. Чтобы настроить параллельную среду, создается новое приложение на основе решения по умолчанию в его нынешнем виде. Это можно сделать в текущей среде разработки песочницы без неблагоприятных последствий для существующей работы.

После завершения шагов в этой статье пользователи с соответствующей ролью увидят ваше новое приложение в списке приложений как в раскрывающемся списке приложений Dynamics 365, так и на домашней странице Dynamics 365 (https://home.dynamics.com).

![Список приложений](media/app-list.png)

После завершения задач, описанных в этом разделе, в среде разработки с использованием решения можно импортировать это решение в среду тестирования, что позволит более широкой группе бизнес-пользователей протестировать и сравнить приложение в знакомой среде. 

Следуйте вашим процессам управлением жизненным циклом приложений (ALM) и операциям разработки. Рекомендуется выполнить все описанные шаги в контексте решения. После тестирования и проверки приложения в среде разработки можно дальше тестировать приложение, запустив его в качестве пилотной программы для большего числа пользователей, например в производственной среде. Наличие управляемого моделью приложения и карты сайта в решении позволяет экспортировать ваше приложение из среды, где вы создали это новое приложение, следуя существующим процессам далее в вашем канале среды. 

## <a name="process-for-validating-your-legacy-web-client-app-in-an-existing-environment"></a>Процедура проверки старого приложения веб-клиента в существующей среде
 
Процесс проверки старого приложения веб-клиента в существующей среде включается три шага:  

1.  Создание нового решения на основе решения по умолчанию
2.  Создание нового управляемого моделью приложения 
3.  Настройка свойств приложения  

Если вы недавно включили для режима **Использовать только единый интерфейс** значение **Вкл.** в своей среде разработки, например в соответствии с инструкциями из темы [Краткое руководство по переводу старого приложения веб-клиента Dynamics 365 на единый интерфейс](transition-web-app.md), необходимо вернуть для этого параметра значение **Откл.**, чтобы можно было запускать существующие старые приложения веб-клиента.

### <a name="create-a-new-solution-thats-based-on-the-default-solution"></a>Создание нового решения на основе решения по умолчанию
1. Вход в [портал PowerApps](https://make.powerapps.com).   
2. Из списка сред выберите нужную среду.  
3. В левой области навигации выберите **Решения**. 
4. В строке меню выберите **Создать решение**. 
5. В области **Создать решение** введите следующие свойства: 
   - **Имя**. Введите название решения. Например, *Приложение единого интерфейса*. 
   - **Издатель**. Выберите издателя, который используется в организации. Обязательно следуйте указаниям по настройке для существующих настроек. Это обеспечит, чтобы имена схем для управляемого моделью приложения и его карты сайта согласованы с существующими стандартами. 
   - **Версия**. Это должно быть установлено в соответствии с существующими стандартами и указаниями для решений. 
6. Выберите **Создать**.  
7. Новое решение создается в списке решений. Выберите его для открытия решения и перейдите к следующему разделу. 

### <a name="create-a-new-model-driven-app-in-the-new-solution"></a>Создание нового управляемого моделью приложения в новом решении
На этом этапе будет создано новое приложение, использующее текущие настройки, чтобы можно было проверить их в едином интерфейсе. Приложение создается в контейнере нового решения, созданного в предыдущем разделе.  

1. В строке меню выберите **Создать**, укажите на **Приложение**, затем выберите **Приложение на основе модели**.
2. На странице **Создать приложение** введите следующие свойства: 
   - **Имя**. Введите название для приложения на свое усмотрение. Например, *Название вашего приложения* + *Новое* или *Тест единого интерфейса*. 
   - **Уникальное имя**. Начинается с префикса решения и упрощенной версии указанного вами имени приложения. Можно внести изменения или оставить его без изменений.  
   - **Описание**. Добавьте описание для приложения, например *Для целей тестирования нового единого интерфейса нашего решения*.  
3. Выберите **Использовать имеющееся решение для создания приложения**, затем выберите **Далее**. 
4. В списке **Выбор решения** выберите **Решение по умолчанию**, в списке **Выбор карты сайта** выберите **Карта сайта**, затем выберите **Готово**.  

   > [!div class="mx-imgBorder"] 
   > ![Выбор существующего решения](media/select-existing-solution.png "Выбор существующего решения")

5. Открывается конструктор приложений, в котором отображаются все компоненты приложения, которые присутствовали в решении по умолчанию. Выберите **Опубликовать**.  
6. После завершения процесса публикации выберите **Воспроизвести**.  

Новое окно открывается в браузере с вашим новым управляемым моделью приложением, которое содержит все объекты, карту сайта, а также настройки карты сайта, которые были в вашем приложении Dynamics 365 по умолчанию.  

> [!div class="mx-imgBorder"] 
> ![Создание приложения единого интерфейса](media/new-unified-interface-app.png "Создание приложения единого интерфейса")

Обратите внимание, что при переходе назад к вкладке браузера с областью **Решения** портала создателя PowerApps ваше новое управляемое моделью приложение и расширение клиента карты сайта с аналогичным названием оба являются частью созданного решения.  

> [!div class="mx-imgBorder"] 
> ![Активы решения](media/solution-assets.png "Активы решения")

На этом шаге вы создали новое управляемое моделью приложение внутри решения, которое можно импортировать в среды тестирования или оценки. Теперь можно начать работать с новым приложением, но перед этим на следующем шаге вы зададите пару настроек для нового приложения. Это сделает возможным предоставление общего доступа к нему другим пользователям.

### <a name="configure-app-properties"></a>Настройка свойств приложения
Задачи, необходимые для настройки свойств управляемого моделью приложения, включают в себя: 

- Назначение ролей пользователей новому приложению, чтобы включить доступ для использования приложения пользователям, не являющимися администраторами.  
- Настройка удобного URL-адреса, чтобы пользователи могли легко делиться ярлыком нового приложения, помещать его в закладки или запоминать его. 


1. Перейдите к *URL-адрес среды*/**apps**. URL-адрес будет выглядеть аналогично следующему: https://*ВашСреда*.crm.dynamics.com/apps. При этом открывается список всех приложений в среде. 
2. Найдите созданное вами новое управляемое моделью приложение.  
3. На плитке приложения выберите многоточие, затем выберите **Управление ролями**.   

   > [!div class="mx-imgBorder"] 
   > ![Управление ролями](media/select-app-ellipsis.png "Управление ролями")

4. Область доступных ролей отображается в правой панели. Выберите требуемые роли, чтобы дать пользователям, не являющимся администраторами, доступ к приложению. 

    > [!IMPORTANT]
    > Убедитесь, что всем пользователям предоставлена по крайней мере одна роль безопасности, которая содержит доступ **Чтение** к привилегии **Приложение на основе модели**. Эту привилегию можно найти на вкладке "Пользовательская настройка" внутри роли безопасности. Пользователи без этой привилегии получают ошибку при открытии любого управляемого моделью приложения.  Обратите внимание, что у ролей безопасности системного администратора и настройщика системы эта привилегия уже включена. 
 
   > [!div class="mx-imgBorder"] 
   > ![Привилегия приложения на основе модели](media/model-driven-app-privilege.png "Привилегия приложения на основе модели")

5. Кроме того, в области **Управление ролями** можно развернуть пункт **Суффикс URL-адреса приложения**, чтобы настроить удобный URL-адрес для управляемого моделью приложения. Обратите внимание, что можно указать почти что угодно. Например, введите *новое*, чтобы в окне предварительного просмотра отображался URL-адрес *https://YourEnvironment.crm.dynamics.com/apps/new*.   

   > [!div class="mx-imgBorder"] 
   > ![Суффикс URL-адреса приложения](media/app-url-suffix.png "Суффикс URL-адреса приложения")

   Это становится удобным URL-адресом для использования и общего доступа, чтобы пользователи могли напрямую попробовать единый интерфейс. Пользователи могут поместить эту ссылку в закладки для удобства. 

6. Нажмите кнопку **Сохранить**. 

Теперь пользователи с соответствующей ролью увидят ваше новое приложение в списке приложений как в раскрывающемся списке приложений Dynamics 365, так и на домашней странице Dynamics 365 (https://home.dynamics.com). 
  
   ![Список приложений](media/app-list.png "Список приложений")

Поскольку для режима **Использовать только единый интерфейс** задано значение **Откл.**, когда пользователи переходят на корневой URL-адрес среды, они все еще попадают на приложение по умолчанию **Dynamics 365 – Custom**, как и ранее. Это ожидается, если они хотят продолжить поддержку существующих старых приложений веб-клиента во время тестирования и работы с приложениями единого интерфейса.  

## <a name="compare-your-new-app-to-the-default-dynamics-365-app"></a>Сравнение вашего нового приложения с приложением Dynamics 365 по умолчанию  
Теперь вы готовы запустить приложение. Можно сравнивать новое приложение единого интерфейса с приложением Dynamics 365 – Custom, используя те же данные, роли пользователей, бизнес-правила, рабочие процессы, подключаемые модули и т. д., поскольку они находятся в той же среде. Это позволяет обучить вашу организацию, поскольку все основные фундаментальные элементы по-прежнему присутствуют, и одновременно, можно начать изучать улучшения нового единого интерфейса.  

> [!TIP]
> Если требуется показать эти приложения рядом на одном мониторе, можно нажать клавишу Windows и клавишу со стрелкой влево (или клавишу со стрелкой вправо), чтобы разделить окна браузера на одном мониторе. 

> [!NOTE]
> Если продолжить выполнять настройки в карте сайта по умолчанию, такие как изменение переходов или более глубокие настройки ленты для кнопок и действий, необходимо повторить процесс, создав другое управляемое моделью приложение или выполнив такие же настройки в новой карте сайта, связанной с вашим управляемым моделью приложением.  

## <a name="validate-your-new-app"></a>Проверка нового приложения  
С помощью приложения, демонстрирующего единый интерфейс, можно запускать проверку приложения, процессов и настроек для определения, как будет выглядеть переход. Рекомендуется протестировать все варианты использования, но можно начать с самых важных или группировать их в логические схемы дизайна. Поскольку единый интерфейс базируется на отзывчивом дизайне, рекомендуется всегда выполнять тесты на различных устройствах с различным разрешением экрана. По мере тестирования приложения вы сможете убедиться, что настройки с совместимы с единым интерфейсом, а также определить, имеются ли какие-либо функции, которые требуют переработки или у которых отсутствует функциональность.  

> [!IMPORTANT]
> Текущая версия Common Data Service и приложения на основе модели в Dynamics 365 все еще содержат ряд устаревших функций. Следует проверить приложение на любые устаревшие функции и при необходимости заменить их новыми возможностями. Дополнительные сведения: [Предстоящие важные изменения (устаревшие функции)](/dynamics365/get-started/whats-new/customer-engagement/important-changes-coming)

> [!TIP]
> Средство проверки PowerApps поможет в проверке качества компонентов решения.  Дополнительные сведения: [Использование средства проверки решений для проверки приложений на основе модели в PowerApps](../common-data-service/use-powerapps-checker.md)

## <a name="next-steps"></a>Дальнейшие шаги
В зависимости от ваших заключений, ваша группа реализации или партнер могут оценить объем работы, которую необходимо выполнить для перевода приложений на единый интерфейс, а также определить возможные улучшения удобства использования. В связи с наличием многочисленных новых функций и возможностей, доступных в едином интерфейсе, имеется возможность увеличить ценность для пользователей приложений. 

Переход на единый интерфейс — это замечательная возможность создать современный интерфейса пользователя и перепроверить существующие процессы, чтобы убедиться, что они все еще правильны или нуждаются в улучшении. Это также удобное время для проверки, отражает ли приложение требования бизнеса и может ли существующее приложение быть распространен на несколько приложений для различных рабочих групп и ролей.
