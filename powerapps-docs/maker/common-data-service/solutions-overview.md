# <a name="solutions-overview"></a>Обзор решений  

  В PowerApps решения используются для транспортировки приложений и компонентов из одной среды в другую или же для применения набора настроек к существующим приложениям. Решение может содержать одно или несколько приложений, а также другие компоненты, например сущности, наборы параметров и т. д.  Вы можете получить решение из [AppSource](https://appsource.microsoft.com/) или от независимого поставщика программного обеспечения.
  
Дополнительные сведения: [Технический документ. Управление жизненным циклом решений](https://www.microsoft.com/en-us/download/details.aspx?id=57777)  
  
> [!NOTE]
>  Если вы являетесь независимым поставщиком программного обеспечения и создаете приложение, которое планируете распространять, вам потребуется использовать решения. Подробные сведения об использовании решений см. в статье [Общие сведения о решениях](/powerapps/developer/common-data-service/introduction-solutions).  
  
 Если вы планируете создавать приложения PowerApps для использования в организации или настраивать Dynamics 365 для приложений Customer Engagement, вам нужно знать о решениях следующее:  
  
-   Создание решений не является обязательным. Вы можете создавать или настраивать приложения в своей среде PowerApps напрямую, без создания решения.  
  
-   При настройке среды PowerApps напрямую без создания решений вы работаете со специальным решением, называемым **решением Common Data Services по умолчанию**. Это решение содержит все настройки вашей среды PowerApps.  
  
-   Кроме того, доступно еще одно специальное решение, называемое **решением по умолчанию**. Это решение содержит все компоненты в вашей системе независимо от того, кто создал их. Вы можете экспортировать **решение по умолчанию**, чтобы создать резервную копию настроек, заданных в вашей организации. Изменения рекомендуется сохранять на случай сбоя.

### <a name="components"></a>Компоненты  
 Компонент представляет собой нечто, что потенциально можно настроить. Все, что можно включить в решение, является компонентом. Ниже приведен список компонентов, которые можно просмотреть в решении:

-   Учреждение  
  
-   Связь сущностей  
  
-   Поле  
  
-   Профиль безопасности поля  

-   Поток

-   Источник данных виртуальной сущности
  
-   Веб-ресурс  
  
 Некоторые компоненты вложены в другие компоненты. Например, сущность содержит формы, представления, диаграммы, поля, отношения сущностей, сообщения и бизнес-правила. Для существования каждому из этих компонентов требуется сущность. Поле не может существовать вне сущности. Мы говорим, что поле зависит от сущности. Фактически существует вдвое больше типов компонентов, чем показано в предыдущем списке, но большинство из них не вложены в другие компоненты и не отображаются в приложении.  
  
 Наличие компонентов необходимо для отслеживания любых ограничений относительно компонентов, которые можно настроить с помощью управляемых свойств, и всех зависимостей, чтобы можно было выполнить экспорт, импорт и (в управляемых решениях) удаление без остатка.  
  
<a name="BKMK_ManagedAndUnmanagedSolutions"></a>   
### <a name="managed-and-unmanaged-solutions"></a>Неуправляемые и управляемые решения  
 Решения могут быть **управляемыми** и **неуправляемыми**. **Управляемое** решение нельзя изменить и можно удалить после импорта. При удалении решения удаляются все его компоненты.  
  
 При импорте **неуправляемого** решения вы добавляете все его компоненты в свою среду. Компоненты невозможно удалить, удалив решение.  
  
 При импорте **неуправляемого** решения, содержащего уже настроенные вами компоненты, ваши настройки будут перезаписаны настройками в импортируемом неуправляемом решении. Это невозможно отменить.  
  
> [!IMPORTANT]
>  Устанавливайте неуправляемое решение только в том случае, если хотите добавить все компоненты в свою среду и перезаписать все существующие настройки.  
  
 Даже если вы не планируете распространять свое решение или настройки, может потребоваться создать и использовать неуправляемое решение для создания отдельного представления, содержащего только те части приложения, которые вы уже настроили. Каждый раз настраивая что-либо, просто добавляйте это в созданное неуправляемое решение.  
  
 **Решение по умолчанию** можно экспортировать только в виде неуправляемого решения.  
  
 Чтобы создать **управляемое** решение, выберите параметр **Как управляемое** при экспорте решения. При создании управляемого решения его невозможно импортировать обратно в ту же среду, которая использовалась для его создания. Его можно импортировать только в другую среду.  
  
<a name="BKMK_HowSolutionsAreApplied"></a>   
### <a name="how-solutions-are-applied"></a>Применение решений  
 Все решения оцениваются как слои, чтобы определить, что именно будет делать приложение. На следующей схеме показано, как оцениваются управляемые и неуправляемые решения и как изменения в них будут выглядеть в вашей среде.  
  
 ![Разделение решения на слои](media/solution-layering.png "Разделение решения на слои")  
  
 Работа в направлении снизу вверх:  
  
 **Системное решение**  
 Системное решение — это управляемое решение, которое присутствует в каждой среде. Системное решение является определением для всех готовых компонентов в системе.  
  
 **Управляемые решения**  
 Управляемые решения могут изменять компоненты системного решения и добавлять новые компоненты. Если установлено несколько управляемых решений, первое из установленных является подчиненным по отношению к управляемому решению, которое было установлено позже. Это означает, что решение, установленное вторым, может настраивать решение, которое было установлено до него. Если два управляемых решения имеют конфликтующие определения, применяется общее правило "побеждает последний". При удалении управляемого решения в силу вступает подчиненное ему управляемое решение. Если удалить все управляемые решения, применяется процедура по умолчанию, определенная в системном решении.  
  
 **Неуправляемые настройки**  
 Неуправляемые настройки — это любые изменения, которые вы внесли в свою среду с помощью неуправляемого решения. Системное решение определяет, что вы можете или не можете настроить с помощью управляемых свойств. Издатели управляемых решений также могут ограничить вашу возможность настраивать компоненты решения, добавленные ими в свое решение. Вы можете настроить любой из компонентов решения, не имеющий управляемых свойств, которые не позволяют вам их настроить.  
  
 **Поведение приложения**  
 Это то, что вы видите в своей среде. Системное решение по умолчанию плюс любые управляемые решения плюс любые неуправляемые настройки, которые вы применили.  
  
<a name="BKMK_ManagedProperties"></a>   
### <a name="managed-properties"></a>Управляемые свойства  
 Некоторые компоненты настроить невозможно. Эти компоненты в системном решении имеют метаданные, которые не позволяют вам их настроить. Они называются **управляемыми свойствами**. Издатель управляемого решения также может задать управляемые свойства, чтобы запретить вам настраивать свое решение теми способами, которые он считает недопустимыми.  
  
<a name="BKMK_Dependencies"></a>   
### <a name="solution-dependencies"></a>Зависимости решения  
 Из-за особенностей размещения управляемых решений слоями некоторые такие решения могут зависеть от компонентов в других управляемых решениях. Некоторые издатели решений будут использовать это, чтобы создавать модульные решения. Вам может потребоваться сначала установить "базовое" управляемое решение, после чего можно будет установить второе управляемое решение, которое дополнительно настраивает компоненты базового решения. Второе управляемое решение зависит от компонентов решения, входящих в первое решение.  
  
 Система отслеживает эти зависимости между решениями. Если вы попытаетесь установить решение, требующее базовое решение, которое не было установлено, то не сможете это сделать. Вы получите сообщение о том, что для использования этого решения нужно сначала установить другое решение. Аналогичным образом, из-за зависимостей вы не можете удалить базовое решение, пока зависимое от него решение все еще установлено. Нужно удалить зависимое решение перед удалением базового решения.
