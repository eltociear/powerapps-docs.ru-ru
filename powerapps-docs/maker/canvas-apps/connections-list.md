# <a name="overview-of-canvas-app-connectors-for-power-apps"></a>Общие сведения о соединителях Canvas-App для Power Apps
Данные являются основой большинства приложений, включая те, которые вы создаете в Power Apps. Эти данные хранятся в *источниках данных*. Вы переносите их в приложение при создании *подключения*. Для подключения используется определенный *соединитель*, который обеспечивает обмен данными с источником данных. В Power Apps есть соединители для многих популярных служб и локальных источников данных, включая SharePoint, SQL Server, Office 365, Salesforce и Twitter. Чтобы приступить к добавлению данных в приложение Canvas, см. статью [Добавление подключения к данным в Power Apps](add-data-connection.md).

Соединитель может предоставлять **таблицы** данных или **действия**. Некоторые соединители предоставляют только таблицы, некоторые — только действия, а некоторые — и то, и другое. Соединитель может быть стандартным или настраиваемым.

## <a name="tables"></a>Таблицы

Если ваш соединитель предоставляет таблицы, вы добавляете источник данных, а затем выбираете таблицу в источнике данных, которой вы хотите управлять. Power Apps извлекает данные таблицы в приложение и обновляет данные в источнике данных. Например, можно добавить источник данных, который содержит таблицу с именем **Lessons**, и задать для свойства **Items** элемента управления (например, коллекции или формы) следующее значение в строке формул:

 ![Свойство Items простого источника данных](./media/connections-list/ItemPropertyPlain.png)

Данные, извлекаемые приложением, можно указать, настроив свойство **Items** элемента управления, отображающего данные. Продолжая предыдущий пример, вы можете сортировать или фильтровать данные в таблице **Lessons** с помощью этого имени в качестве аргумента для функций **Search** и **SortByColumn**. На этом рисунке формула, по которой задано свойство **Items**, указывает, что данные сортируются и фильтруются в соответствии с текстом в поле **TextSearchBox1**. 

 ![Развернутое свойство Items источника данных](./media/connections-list/ItemPropertyExpanded.png)

## <a name="actions"></a>Действия

Если ваш соединитель предоставляет действия, вам также нужно выбрать источник данных. Но вместо выбора таблицы в качестве следующего шага, вам нужно вручную подключить элемент управления к действию путем изменения свойства **Items** элемента управления, который будет отображать данные. Формула, для которой вы задаете свойство **Items**, указывает действие, которое извлекает данные. Например, приложение не получит никакие данные, если вы подключитесь к Yammer, а затем укажете для свойства **Items** имя источника данных. Чтобы заполнить элемент управления данными, укажите действие, например **GetMessagesInGroup(5033622).messages**.

![Свойство Items для действия источника данных](./media/connections-list/ItemPropertyAction.png)

Если вам нужно обрабатывать пользовательские обновления данных для соединителей действия, создайте формулу с функцией **Patch**. В формуле определите действие и поля, которые требуется привязать к действию.  

Дополнительные сведения о настройке формулы для пользовательских обновлений см. в следующих разделах:

[Patch](functions/function-patch.md)<br>[Collect](functions/function-clear-collect-clearcollect.md)<br>[Update](functions/function-update-updateif.md)

> [!NOTE]
>  **Power Apps не работает с динамической схемой**. Динамическая схема фразы означает возможность того, что одно и то же действие может вернуть другую таблицу с разными столбцами. Условия, которые могут привести к различиям в столбцах в таблицах, включают входные параметры действия, пользователя или роль, которая исполняет действие, а также группу, в которой работает пользователь, и т. д. Например, SQL Server хранимые процедуры могут возвращать различные столбцы, если они выполняются с разными входными данными. Для действий с динамической схемой документация по соединителю показывает, **что выходные данные этой операции являются динамическими.** в качестве возвращаемого значения. В отличие от этого, Power автоматизиру работает с динамической схемой и может предоставить обработку для вашего сценария.

## <a name="security-and-types-of-authentication"></a>Безопасность и типы проверки подлинности

При создании приложения и подключении к источнику данных вы можете увидеть, что выбранный соединитель позволяет использовать разные способы проверки подлинности. Например, соединитель SQL Server позволяет использовать интегрированную службу Azure AD, SQL Serverную проверку подлинности и проверку подлинности Windows. Каждый тип проверки подлинности имеет разные уровни безопасности, связанные с ней.  Важно понимать, какие сведения и права совместно используются с пользователями вашего приложения. Основной пример в этой статье — SQL Server, однако принципы применяются ко всем типам соединений.

### <a name="azure-ad-integrated"></a>Интегрированная служба Azure AD

Это защищенный тип соединения.  Например, в SharePoint используется такой тип проверки подлинности.  SQL Server также поддерживает этот тип проверки подлинности.  При подключении служба Azure AD идентифицирует вас отдельно для SharePoint от вашего имени.  Указывать имя пользователя или пароль не требуется.  Как автор вы можете создать источник данных и работать с ним с помощью своих учетных данных.  При публикации приложения и входе пользователя приложения в систему они получают свои учетные данные. Если данные надлежащим образом защищены на внутреннем сервере, пользователи смогут видеть только те, которые им разрешено просматривать на основе их учетных данных.   Этот тип безопасности позволяет изменять права для конкретных пользователей приложения в источнике данных серверной части после публикации приложения.  Например, можно предоставить доступ, запретить доступ или уточнить, что пользователь или набор пользователей могут видеть на внутреннем источнике данных.

### <a name="open-standard-authorization-oauth"></a>Открытие — стандартная авторизация (OAuth)

Этот тип подключения также является безопасным.  Например, в Twitter используется такой тип проверки подлинности.  При подключении необходимо указать имя пользователя и пароль.  Как автор вы можете создать источник данных и работать с ним с помощью своих учетных данных.  При публикации приложения и входе пользователя приложения в систему они должны также предоставить свои учетные данные.  Поэтому такой тип подключения безопасен, так как пользователи должны использовать собственные учетные данные для доступа к службе источника данных. 

### <a name="sql-user-name-and-password-authentication"></a>Имя пользователя SQL и проверка подлинности по паролю

Это тип подключения не очень безопасен, так как не зависит от проверки подлинности пользователей.  SQL Server также поддерживает этот тип проверки подлинности.  В SQL Server такой тип проверки подлинности называется **SQL Server аутентификация**.  Многие другие источники данных базы данных предоставляют аналогичные возможности.  При публикации приложения пользователям не нужно указывать уникальные имя пользователя и пароль.  Они используют имя пользователя и пароль, которые указываются при создании приложения.  Проверка подлинности подключения к источнику данных **неявно** предоставляется пользователям.  После публикации приложения подключение также публикуется и становится доступным для пользователей.  Конечные пользователи также могут создавать приложения, используя любое подключение, используя проверку подлинности SQL Server, к которой они имеют доступ.  Пользователи не видят имя пользователя Password, но подключение будет доступно для них.  Для этого типа подключения вполне вполне допустимое количество сценариев.  Например, если у вас есть база данных только для чтения, доступная всем сотрудникам компании, этот тип подключения может быть допустимым. 

### <a name="windows-authentication"></a>Проверка подлинности Windows

Это тип подключения не очень безопасен, так как он также не зависит от проверки подлинности пользователей.  Проверка подлинности Windows используется, если необходимо подключиться к **локальному**источнику данных.  Примером такого типа подключения является локальный сервер с SQL Server.  Подключение должно проходить через шлюз.  Так как он проходит через шлюз, у соединителя есть доступ ко всем данным в этом источнике данных. В результате все сведения, к которым можно получить доступ с помощью предоставленных учетных данных Windows, доступны соединителю. После публикации приложения подключение также публикуется и становится доступным для пользователей.   Это означает, что конечные пользователи могут также создавать приложения, использующие такое же подключение и получать доступ к данным на этом компьютере.  Подключения к источнику данных также **неявно** предоставляются пользователям, к которым предоставлен общий доступ к приложению.  Этот тип подключения может быть допустимым, если источник данных находится только на локальном сервере, и данные в этом источнике свободно являются общими.
