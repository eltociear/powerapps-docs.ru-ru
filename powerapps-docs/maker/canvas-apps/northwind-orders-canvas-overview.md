# <a name="overview-of-the-canvas-app-for-northwind-traders"></a>Обзор приложения canvas для Northwind Traders

Дополнительные сведения о приложении canvas для управления реляционными данных в базе данных Northwind Traders базы данных, которые [установлен в вашей среде](northwind-install.md). Затем выполните пошаговые инструкции в последующих разделах, чтобы создать такое приложение с нуля, таким образом, получают практический опыт работы с реляционными данными.

В этом разделе обнаружения:

- Как пользователя приложения показано и управление реляционными данными в приложении.
- Какие типы данных управления приложением.
- Как создаются связи между этими типами данных.

На одном экране пользователь приложения можно показать, обновление, создание и удаление заказов.

## <a name="explore-the-user-interface"></a>С пользовательским интерфейсом

### <a name="order-gallery"></a>Порядок коллекции

У левого края, приложения в коллекции отображаются список заказов, включая порядковый номер, состояние, имя клиента и общую стоимость заказа. Пользователь может перемещаться по списку, чтобы найти заказ, а затем Показать дополнительные сведения о нем, щелкнув стрелку заказа. Дополнительные сведения: [Создание коллекции порядок](northwind-orders-canvas-part1.md).

### <a name="summary-form"></a>Сводка формы

В правом верхнем углу формы приведена сводка заказа, что пользователь выбрал в коллекции заказа. Сводка содержит большую часть этой информации как выполняет этой коллекции, но в сводке показано также даты, когда порядок был создан и оплачен, а также имя и изображение сотрудника, управляемые порядок. Пользователь может изменять данные в форме, сохранить эти изменения, отменить их или удалить заказ, выбрав значок рядом с правым краем строки заголовка. Дополнительные сведения: [Создание сводной форме](northwind-orders-canvas-part2.md).

### <a name="detail-gallery"></a>Сведения о коллекции

В правом нижнем углу в другой коллекции отображаются сведения о какие продукты, содержащий выбранный заказ и в какие количества. Каждый элемент в коллекции называется подробности заказа. Пользователь приложения можно добавлять и удалять любой элемент в этой коллекции с помощью элементов управления в и под ним. Дополнительные сведения: [Создание коллекции сведений](northwind-orders-canvas-part3.md).

## <a name="explore-the-data-sources"></a>Просматривать источники данных

Чтобы создать это приложение, вы покажете, данные из пяти сущностей и набора параметров. На самом деле большинство областей это приложение отображает данные из нескольких сущностей. Например порядок коллекции содержат следующие сведения:

- Порядковый номер — это поле в **заказы** сущности.
- Другое поле находится в состоянии **заказы** сущности, параметр из **состояния заказов** набор параметров.
- Имя клиента — это поле в **клиентов** сущности.
- Общая стоимость рассчитывается на основе данных в **Order Details** сущности.

Сводка содержит некоторые те же сведения, как и список заказов, но также содержит имя и изображение сотрудника, управляемые порядок. Что извлекается из полей в **сотрудников** сущности. В коллекции сведений отображаются записи в **Order Details** сущности и каждый продукт в эти сведения — это запись в **заказ продуктов** сущности.

## <a name="explore-the-relationships"></a>Исследования связей

Можно показать данные из различных источников (например, сущности) в одной коллекции или форме, так как эти сущностями есть связи, которые были созданы для вас в базе данных.

### <a name="many-to-one-relationships"></a>Многие к одному связей

Например, сведения о клиенте и сотрудника для каждого заказа находятся в **клиентов** и **сотрудников** сущностей. Таким образом **заказы** сущности имеются связи многие к одному с этими сущностями, так как существует много заказов, каждый из которых можно только одного клиента и управляет только один сотрудник.

Также, каждый заказ имеет один или несколько элементов строки, представляющие продукты, которые содержит порядок и их количества. Каждый элемент строку — это запись в **Order Details** сущность, которая извлекает информацию о каждом продукте из **заказ продуктов** сущности. Каждой дополнительных сведений идентифицирует только один продукт, но каждый товар может присутствовать несколько подробно. Таким образом **Order Details** сущность имеет отношение "многие к одному" с **заказ продуктов** сущности.

### <a name="one-to-many-relationships"></a>Один ко многим связей

Каждый заказ может содержать несколько элементов строки, но каждый элемент строку относится к только один заказ. Таким образом **заказы** сущность имеет отношение "один ко многим" с **Order Details** сущности.

### <a name="dot-notation-for-relationships"></a>Нотация с точками для связей 

Для отображения данных на основе связи между сущностями, можно использовать свойство точку-разделитель для прохода по связи от одной сущности к другому.  Например, каждая запись в **заказы** сущности извлекает сведения из **клиентов** сущности, что порядок коллекции отображать имена клиентов. В этой коллекции, это поведение настраивается путем установки **текст** метки — это выражение:<br>`ThisItem.Customer.Company`

**ThisItem** указывает запись в **заказы** сущности и переносит сведения из **клиентов** сущностью в отношении клиента, разместившего заказ. В этом случае выражение указывает, что отображается название компании заказчика. Тем не менее вся запись для этого клиента будут извлечены, поэтому можно легко показать, к примеру, адрес электронной почты для этого клиента вместо этого.

Другой пример переходит от одной сущности к другой можно указать, что коллекции должны отображаться записей в одной сущности, на основе записи, и пользователь выбрал в другой коллекции в другую сущность. Чтобы отобразить сведения о заказе, необходимо настроить галерею подробно **элементы** свойство следующее выражение:<br>`Gallery1.Selected.'Order Details'`

В этом случае **Gallery1.Selected** указывает запись в **заказы** сущности, так же, как **ThisItem** как в предыдущем примере. Тем не менее это выражение не извлекать только одну запись, как предыдущее выражение. Вместо этого он извлекает всю таблицу записей для отображения имени и удельные затраты каждого продукта (что отражено в **заказ продуктов** сущности) и количество (как отражено в **Order Details** сущности).
